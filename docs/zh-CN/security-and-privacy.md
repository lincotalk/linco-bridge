# 安全与隐私

[English](../security-and-privacy.md)

本页从仓库级别总结 Linco Bridge 的安全与隐私边界。它比连接器安全说明更轻量，但也应该让外部读者先理解哪些内容必须留在本机、哪些说法不能轻易对外承诺，以及哪些链路在接真实数据前必须额外审查。

## 核心安全模型

Linco Bridge 连接的是远端客户端和用户自己电脑上的 Agent。因此它的安全边界不只是网络传输，还包括：

- 本地凭证如何保存；
- 文件读取和文件下发如何受限；
- 危险操作如何确认；
- 不同通道下的数据可见性；
- 连接器、后端、客户端 UI 之间的信任边界。

## 项目级原则

- 不要提交 App Secret、Token、私钥、用户数据或带 token 的本地测试页地址。
- 官方通道、自托管参考平台、在线 Demo 和自定义 adapter 的数据可见性并不相同，连接真实数据前应先审查实际链路。
- 本地 `linco-demo` 默认使用 `ws://127.0.0.1:3300`，只适合开发验证；公网部署应配置 TLS/WSS、鉴权、存储、审计和日志脱敏。
- TLS/WSS 不等于端到端加密。未针对实际链路完成实现与审查前，不宣称端到端加密、零知识或完全本地处理。
- 官方服务和生产后端必须把连接器 frame 视为不可信输入，包括来自修改版或 fork 客户端的 frame。

## 不同部署路径的隐私差异

| 路径 | 典型隐私边界 | 说明 |
| --- | --- | --- |
| 官方通道（`linco`） | 跟随官方产品链路 | 以官方产品实际部署和政策为准 |
| 本地参考平台（`linco-demo`） | 多数数据停留在用户本机和本地服务 | 适合验证，不等于生产安全模型 |
| 官方在线 Demo | 公共体验环境 | 如果没有显式实现隔离，不应把它当成私有、长期留存空间 |
| 自定义通道 | 取决于你自己的后端与客户端设计 | 鉴权、存储、审计、删除策略都由你负责 |

## 文件与附件边界

生成文件和附件能力很强，但也是安全面：

- 文件下发应限制在允许目录内、经过校验的非隐藏文件；
- `.env`、`.git/config`、`.ssh/*` 等隐藏文件默认应继续拦截；
- 日志不应输出完整 secret、完整认证 URL 或附件 base64；
- 公共 Demo 截图和示例中应避免暴露个人数据、本机路径或运营资产。

## 审批与危险操作

仓库文档应继续清晰说明审批模型：

| 模式 | 含义 |
| --- | --- |
| `manual` | 权限请求和危险操作都需要显式确认 |
| `auto` | 自动确认，但仍保留默认权限边界 |
| `yolo` | 尝试使用 Agent 原生绕过模式，只适合可信环境 |

如果某条部署路径、某个 UI 流程或某份示例文档改变了默认审批行为，也应明确写出来。

## 不应轻易对外承诺的说法

以下说法只有在对应链路真实实现并验证后才能写：

- 端到端加密；
- 零知识存储；
- 完全本地处理；
- 公共 Demo 天然具备持久化隐私隔离；
- 生产级多租户安全承诺。

## 贡献者审查清单

当文档或代码涉及安全敏感区域时，至少应检查：

- 凭证保存与日志脱敏；
- 文件下发限制与隐藏路径规则；
- 审批流是否变化；
- 公共 Demo 的隔离或共享状态行为；
- 后端是否把连接器上送 frame 当作不可信输入处理。

安全漏洞仍应按 [SECURITY.md](../../SECURITY.md) 走私密报告流程。
